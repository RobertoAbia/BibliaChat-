# **tickets.md — Biblia Chat (MVP)**

**Arquitectura:** Supabase-first (Postgres \+ Auth \+ RLS) \+ Edge Functions para IA/Billing/operaciones sensibles
 **App:** Flutter iOS/Android
 **Monetización:** Paid-first con trial 3 días (mensual)
 **Chat:** múltiples hilos por usuario, 1 hilo por topic en MVP
 **IA:** OpenAI GPT-5.2 stateless \+ `user_profiles.ai_memory` \+ `chats.context_summary`
 **Última actualización:** 02-02-2026

### **Arquitectura Técnica de Monetización (referencia)**

**Edge Functions requeridas:**

| Función | Ticket | Descripción |
|---------|--------|-------------|
| `handle-purchase` | T-0403 | Guardar `rc_app_user_id` en user_profiles al comprar |
| `restore-purchases` | T-0403b | Migrar datos del usuario antiguo al nuevo |
| `revenuecat-webhook` | T-0404 | Recibir eventos de compra/cancelación/renovación |

**Flujo de COMPRA (usuario anónimo):**
```
Flutter → RevenueCat SDK → Compra exitosa en App Store/Google Play
       → Llama Edge Function "handle-purchase"
       → Guarda rc_app_user_id en user_profiles
       → Usuario es premium (sin registro)
```

**Flujo de RESTAURACIÓN (tras reinstalar):**
```
Flutter → RevenueCat SDK → "Restore purchases"
       → Obtiene rc_app_user_id del receipt (vinculado a Apple ID/Google)
       → Llama Edge Function "restore-purchases"
       → Edge Function:
           1. Busca perfil anterior con ese rc_app_user_id
           2. Si existe: migra chats, progreso, badges, etc. al usuario actual
           3. Actualiza rc_app_user_id en perfil actual
       → Usuario recupera premium + TODOS sus datos
       → Experiencia transparente: solo pulsó "Restaurar compras"
```

**Datos a migrar en restore-purchases:**
- `user_profiles` (perfil, ai_memory, preferencias)
- `chats` + `chat_messages` (historial completo)
- `saved_messages` (mensajes guardados)
- `user_plans` + `user_plan_days` (progreso en planes)
- `daily_activity` (rachas)
- `user_badges` + `user_points` (gamificación)

### **Feature: Navegación Android (Back Button + Swipe)** ✅

**Prioridad:** P1
**DoD:**

* ✅ Swipe entre tabs: PageView con `PageScrollPhysics` para deslizar entre Home, Chat, Estudiar y Perfil
  * La pantalla sigue el dedo y encaja naturalmente al soltar
  * Animación de NavigationBar: 300ms con `easeOutCubic`
  * Iteraciones: BouncingScrollPhysics → ClampingScrollPhysics → GestureDetector+NeverScrollableScrollPhysics → PageScrollPhysics (final)
* ✅ Botón atrás Android con comportamiento correcto:
  * En Stories (`/home/stories`) → Vuelve a Home
  * Dentro de un chat → Vuelve a lista de chats
  * En lista de chats → Va a Home
  * En Home → Cierra la app
* ✅ Bottom nav oculto condicionalmente para experiencia fullscreen:
  * En chat: `location.startsWith('/chat/')`
  * En Stories: `location == '/home/stories'`
* ✅ `BackButtonInterceptor` + GoRouter con `context.push()`
* ✅ **IMPORTANTE**: Rutas fullscreen deben estar DENTRO del ShellRoute (para que `currentLocationProvider` se actualice)
* ✅ **IMPORTANTE**: NUNCA usar `Navigator.push()` para rutas de GoRouter (bypasea el router)
* ✅ Documentación técnica: `docs/back-button-intentos.md` (historial completo + solución final)

**Solución técnica para Chat:**
- La causa raíz era que `chat_list_screen.dart` usaba `Navigator.push()` en lugar de `context.push()`
- Cambiar a `context.push('/chat/id/xxx')` permite que GoRouter maneje correctamente la navegación

**Solución técnica para Stories:**
- La causa raíz era que `/stories` estaba FUERA del ShellRoute
- `MainShell.build()` no se ejecutaba → `currentLocationProvider` no se actualizaba

### **Fix: Racha no se incrementa al ver 3 Stories** ✅

**Prioridad:** P0
**DoD:**

* ✅ Después de ver las 3 stories, la racha se incrementa correctamente (de 0 a 1)
* ✅ El SnackBar de celebración muestra el valor correcto
* ✅ No hay race condition en el sistema de Optimistic UI

**Problema:**
- Después de ver las 3 stories, la racha mostraba 0 en lugar de 1
- Solo apareció después de implementar el aislamiento de datos por usuario

**Causa raíz:**
- Race condition en el sistema de "Optimistic UI" para la racha
- `markDayAsCompleted()` usaba `Future.delayed(500ms)` para limpiar el estado optimista
- Si la recarga de Supabase tardaba más de 500ms, el estado optimista se limpiaba antes de que llegaran los datos reales
- El provider quedaba en estado "loading" y `.valueOrNull ?? 0` retornaba 0

**Por qué apareció ahora:**
- El aislamiento de datos por usuario añadió `ref.watch(currentUserIdProvider)` a `streakDaysProvider`
- Esto añade unos milisegundos extra al re-evaluar la cadena de dependencias
- Los 500ms ya no eran suficientes

**Solución:**
- Reemplazar `Future.delayed(500ms)` por `await ref.read(streakDaysProvider.future)`
- Esto asegura que el estado optimista se limpia SOLO después de que los datos reales han cargado

**Archivo modificado:**
* `lib/features/home/presentation/providers/daily_progress_provider.dart`

### **Fix: Lista de chats no se actualiza al volver de un chat** ✅

**Prioridad:** P1
**DoD:**

* ✅ Al crear una nueva conversación y volver al tab de chats, la conversación aparece en la lista
* ✅ La lista se actualiza automáticamente sin necesidad de pull-to-refresh
* ✅ Funciona con ShellRoute (widgets montados permanentemente)

**Problema:**
- Al crear una nueva conversación y volver al tab de chats, la conversación no aparecía en la lista
- Incluso navegando a otros tabs y volviendo, la lista no se actualizaba
- Solo funcionaba con pull-to-refresh manual

**Intentos fallidos:**
- Route listeners (`router.routerDelegate.addListener`) - No detectaban correctamente el cambio
- `didChangeDependencies` - No se dispara porque ShellRoute mantiene el widget montado
- Callbacks después de navegación - El widget podía estar disposed

**Causa raíz:**
- Con ShellRoute, `ChatListScreen` se mantiene montado durante toda la navegación
- Los lifecycle methods no detectan cuando el usuario "vuelve" porque el widget nunca se desmonta
- La lista no se enteraba de que había un nuevo chat

**Solución:**
- `ChatNotifier` ahora recibe `Ref` en el constructor
- Añadido método `_notifyChatListRefresh()` que incrementa `userChatsRefreshProvider`
- Después de enviar un mensaje exitosamente, se llama `_notifyChatListRefresh()`
- `ChatListScreen` observa `refreshableUserChatsProvider` (que depende del contador) y se actualiza automáticamente
- **Patrón:** Comunicación directa entre providers en lugar de detectar navegación
